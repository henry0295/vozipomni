Informe DevOpsInfraestructura5/10CI/CD1/10Observabilidad3/10Resiliencia4/10Escalabilidad3/10Veredicto: La infraestructura funciona en mono-nodo pero no está lista para producción empresarial. La ausencia de CI/CD, el uso global de network_mode: host, la falta de secrets management y la nula estrategia de escalabilidad horizontal representan deuda técnica crítica que debe abordarse antes de ofrecer el servicio a clientes.1.1 Infraestructura y Contenedores  Puntos positivos✓ BIENHealthchecks y depends_on bien configuradosTodos los servicios tienen healthcheck con condition: service_healthy. La cadena PostgreSQL → Redis → Backend → Celery está correctamente modelada. Esto es una buena práctica que muchos proyectos omiten.✓ BIENYAML anchors reducen duplicaciónUso de x-django-common, x-logging, x-restart-policy demuestra madurez en la gestión de Compose. El logging JSON con max-size: 10m, max-file: 3 es correcto para entornos de producción.  Hallazgos críticos✗ CRÍTICOnetwork_mode: host en TODOS los servicios de producciónEl docker-compose.prod.yml aplica network_mode: host a PostgreSQL, Redis, Django, Nginx y Celery — no solo a los servicios VoIP. Esto elimina el aislamiento de red entre contenedores, expone puertos internos al host y hace imposible correr múltiples instancias en el mismo nodo. Solo Asterisk, Kamailio y RTPEngine lo necesitan por razones de rendimiento RTP.✗ CRÍTICOSecretos sin gestión centralizadaLas variables sensibles se generan con openssl rand y se guardan en credentials.txt y .env en disco sin cifrado. No hay integración con Docker Secrets, Vault ni AWS Secrets Manager. Si el servidor se compromete, todos los secretos quedan expuestos en texto plano.  Hallazgos de prioridad alta⚠ ALTOAsterisk sin límite de memoria en producciónEl servicio asterisk en docker-compose.prod.yml no tiene mem_limit. Bajo carga de llamadas concurrentes puede consumir toda la RAM del host y derribar PostgreSQL y Django por OOM (Out of Memory).⚠ ALTOVersiones sin pinear en imágenes de telefoníaAsterisk y Kamailio se usan sin versión específica en el stack. Builds no deterministas pueden introducir incompatibilidades entre despliegues sin aviso previo. Se requieren tags semánticos fijos (asterisk:20.5, no asterisk:latest).1.2 Pipeline CI/CD  Hallazgos críticos✗ CRÍTICOAusencia total de CI/CDNo existe ningún archivo en .github/workflows/. El proyecto tiene 298 commits sin un solo pipeline de validación automática. Cualquier push a main puede desplegarse sin lint, tests ni análisis de seguridad.✗ CRÍTICOBranch único sin protecciónSolo existe la rama main. No hay reglas de protección de rama, revisión obligatoria de PRs ni política de merge. Esto hace imposible implementar trunk-based development de forma segura.⚠ ALTODeploy manual sin versionado de artifactsEl flujo de despliegue (deploy.sh, quick-deploy.sh) es completamente manual. No hay versionado de imágenes Docker, no hay rollback automático y no hay registro de qué versión está corriendo en producción.Pipeline CI/CD recomendado (GitHub Actions)Se propone un pipeline de 5 pasos ejecutado en cada PR y merge a main:Paso 1 — Lint & calidad: ruff (Python), ESLint + vue-tsc (Vue/TS), shellcheck (bash). Tiempo estimado: ~2 min.Paso 2 — Tests automatizados: pytest-django para el backend, vitest para el frontend. Objetivo: 40% cobertura mínima en rutas críticas.Paso 3 — Seguridad: bandit (SAST Python), trivy (scan de imágenes Docker), gitleaks (detección de secretos en commits).Paso 4 — Build & push: docker buildx con tags semánticos (:main-sha123). Push a GHCR con firma de imagen.Paso 5 — Deploy a staging con smoke test HTTP, promoción manual a producción. Rollback automático si el healthcheck falla.1.3 Observabilidad y Monitoreo✓ BIENSentry SDK integradoEl backend incluye el SDK de Sentry para captura de errores. Es un buen punto de partida para observabilidad de aplicación.✗ CRÍTICOSin stack de métricas (Prometheus/Grafana)No hay exporters de métricas para Django, Redis, PostgreSQL ni Asterisk. Es imposible detectar degradación de rendimiento, picos de llamadas o saturación de recursos antes de que ocurra un incidente.✗ CRÍTICOSin agregación centralizada de logsLos logs están en contenedores Docker con rotación local (max-size: 10m). No hay ELK stack, Loki ni ningún sistema de búsqueda. Diagnosticar incidentes en producción requiere acceso SSH directo al servidor.1.4 Escalabilidad y Alta Disponibilidad✗ CRÍTICOArquitectura mono-nodo sin HATodo corre en un único servidor. Una falla de hardware significa downtime total del contact center. Para un CCaaS el SLA mínimo es 99.9% (~8.7 horas de downtime/año) — imposible garantizar con nodo único.✗ CRÍTICOPostgreSQL sin réplica ni failoverLa base de datos no tiene réplica de lectura ni failover automático. Una corrupción de datos o falla de disco sin backup reciente implica pérdida irreversible de historial de llamadas, grabaciones y configuración.1.5 Plan de Acción DevOpsPrioridad 1 — Implementar GitHub Actions con lint + trivy + gitleaks en todos los PRs a main. Tiempo estimado: 1-2 días.Prioridad 2 — Restringir network_mode: host solo a Asterisk, Kamailio y RTPEngine. Resto usa bridge network. Tiempo: 4 horas.Prioridad 3 — Integrar Docker Secrets o Vault para gestión de credenciales. Eliminar .env en disco con secretos en texto plano. Tiempo: 1 semana.Prioridad 4 — Pinear versiones de todas las imágenes Docker. Agregar mem_limit: 2g a Asterisk. Tiempo: 2 horas.Prioridad 5 — Añadir contenedor Prometheus + Grafana con exporters (django-prometheus, redis_exporter, postgres_exporter, asterisk_exporter). Tiempo: 2-3 días.Prioridad 6 — Configurar réplica PostgreSQL de solo lectura y Redis Sentinel para HA mínima. Tiempo: 3-5 días.Prioridad 7 — Implementar versionado semántico de imágenes y pipeline de deploy con validación y rollback automático. Tiempo: 1 semana.2. Informe QA (Calidad de Software)Cobertura de tests0%Calidad de código5/10Documentación API8/10Gestión de bugs1/10Testabilidad4/10Veredicto: El proyecto tiene 0% de cobertura de tests automatizados en un sistema de Contact Center donde errores en marcadores predictivos, colas ACD o WebRTC significan llamadas perdidas en producción. No existe ningún proceso de QA estructurado en 298 commits de historial.2.1 Estado del Testing  Hallazgos críticos✗ CRÍTICOCero tests en todo el repositorioNo hay ningún directorio tests/, spec/ ni archivo de configuración de testing (pytest.ini, vitest.config.ts, jest.config.js). 298 commits sin un solo test automatizado en ninguna capa del sistema.✗ CRÍTICOLógica crítica de negocio sin validaciónEl Dialer Engine (marcador predictivo con ratio dinámico 1.0x-3.0x) y la máquina de estados de llamadas (queued→dialing→ringing→answered→completed) son las funcionalidades más críticas. Un bug en el ratio puede disparar llamadas masivas o no marcar nunca — sin ningún test que lo detecte.✗ CRÍTICOSin tests de integración para endpoints RESTLos endpoints de autenticación JWT, campañas y colas ACD no tienen tests. Un cambio de serializer puede romper el contrato de API sin detección automática y afectar a todos los clientes integrados.  Hallazgos de prioridad alta⚠ ALTOSin tests de carga o stressNo hay pruebas de carga documentadas (Locust, k6, JMeter) para validar el comportamiento con 100 agentes concurrentes y 500 llamadas activas. Los cuellos de botella se descubren en producción con clientes reales.⚠ ALTOWebSocket sin tests de integraciónLos eventos en tiempo real del dashboard y la consola de agente son críticos para la operación. No hay tests que validen que los eventos de Asterisk AMI se transmiten correctamente vía Redis PubSub al cliente WebSocket.2.2 Calidad y Mantenibilidad del Código  Puntos positivos✓ BIENSeparación de responsabilidades claraEl uso de composables en Nuxt (useApi, useAuth, useCalls) y la separación backend/frontend/dialer_engine/websocket_server demuestra buena arquitectura. La lógica de negocio no está mezclada con la presentación.✓ BIENTypeScript en el frontendEl frontend usa TypeScript (5.5% del código), lo que reduce errores de tipo en tiempo de desarrollo. Sin embargo, sin un pipeline CI que ejecute vue-tsc --noEmit, el type-checking no se aplica de forma consistente.  Hallazgos de prioridad alta⚠ ALTOSin linter configurado en CINo hay configuración de ruff/flake8 para Python ni ESLint para Vue/TS en pipeline automatizado. La calidad del código depende exclusivamente de la disciplina del desarrollador.⚠ ALTODialer Engine como módulo sin testsEl motor de discado basado en panoramisk maneja múltiples modos (manual, progresivo, predictivo, blasting). Sin tests, cualquier cambio en el ratio predictivo es un cambio a ciegas en producción con impacto directo en operaciones del call center.2.3 Calidad de API y Contrato✓ BIENSwagger/OpenAPI con drf-spectacularLa documentación automática en /api/docs/ es un activo importante. Permite que equipos de QA hagan pruebas exploratorias y que se generen clientes SDK automáticamente.⚠ ALTOSin versionado de APILos endpoints son /api/campaigns/, no /api/v1/campaigns/. Un cambio breaking en los serializers romperá todos los clientes integrados sin período de deprecación ni aviso.⚠ ALTOSin contract testingNo hay tests de contrato (Pact o similar) entre el frontend Nuxt y el backend Django. Los composables (useApi) pueden llamar a endpoints que han cambiado de schema sin que ningún test lo detecte.2.4 Plan de Acción QAPrioridad 1 — Crear backend/tests/ con pytest-django. Informe CCaaS (Contact Center as a Service)Madurez de producto5/10Multi-tenancy1/10Billing & métricas0/10SLA operacional2/10Competitividad5/10Veredicto: VoziPOmni tiene las funcionalidades técnicas correctas de un Contact Center (ACD, IVR, marcadores, WebRTC) pero le faltan los pilares fundamentales para comercializarse como CCaaS: multi-tenancy, billing, SLAs garantizados y onboarding self-service. Hoy es una solución on-premise bien documentada, no un servicio en la nube.4.1 Funcionalidades de Contact Center  Capacidades destacadas✓ FUERTEStack de marcadores completo y competitivoManual, progresivo, predictivo (ratio dinámico 1.0x-3.0x con target de 3% de abandono) y call blasting. Cubre el 90% de los casos de uso de contact centers outbound. El marcador predictivo con ajuste dinámico es una implementación técnicamente sólida.✓ FUERTEWebRTC nativo con JsSIP — sin hardware adicionalLa consola de agente con softphone WebRTC integrado elimina la necesidad de hardware físico o licencias de softphone (Zoiper, Xlite). Diferenciador importante para segmentos que buscan TCO bajo.✓ BIENColas ACD con 6 estrategias de distribuciónringall, leastrecent, fewestcalls, random, rrmemory y linear. Más opciones que muchas plataformas que cobran distribución inteligente como add-on premium.✓ BIENReportería con exportación PDF, Excel, CSV y JSONLos analistas de contact center necesitan exportar reportes para sus sistemas de BI corporativos. Tener múltiples formatos nativos y programación automática de reportes es un punto diferenciador.  Brechas de producto⚠ CRÍTICOCanales digitales ausentes — nombre omnicanal es aspiracionalEl nombre del repositorio dice omnicanal pero hoy es exclusivamente voz. No hay WhatsApp, chat web, email ni SMS integrado. Competidores como Twilio Flex, Genesys Cloud y Talkdesk son verdaderamente omnicanal. El roadmap menciona WhatsApp pero sin fecha de entrega.ℹ MEDIOIVR configurable desde UI pero sin IA conversacionalLos menús de voz interactivos son configurables desde la interfaz, lo que es positivo. Sin embargo, el mercado CCaaS 2026 espera IVR con NLU (Natural Language Understanding) y voicebots. Esto está en roadmap como 'Chatbot con IA' sin fecha.4.2 Multi-tenancy — Brecha Crítica para CCaaS✗ CRÍTICOArquitectura mono-tenant — imposible ofrecer como servicio compartidoEl sistema no tiene ningún concepto de 'tenant' u 'organización' en el modelo de datos. Todos los agentes, campañas y llamadas comparten la misma instancia de BD sin aislamiento. Para CCaaS necesita: schema isolation (BD por tenant) o columna tenant_id en todas las tablas con row-level security en PostgreSQL.✗ CRÍTICOSin onboarding self-servicePara CCaaS, cada nuevo cliente debe poder registrarse, configurar su tenant, añadir agentes y hacer una llamada de prueba sin intervención humana. Hoy el despliegue requiere acceso SSH al servidor y ejecución manual de scripts bash.✗ CRÍTICOAsterisk compartido — riesgo de aislamiento entre clientesEn un modelo multi-tenant, todos los tenants comparten el mismo Asterisk. Esto crea riesgos de: exposición de extensiones SIP entre clientes, acceso a grabaciones de otros tenants, y un cliente que satura el sistema afecta a todos los demás.4.3 Modelo de Negocio CCaaS✗ CRÍTICOSin módulo de billing ni métricas de usoNo hay ningún sistema de medición de uso (minutos de llamada, número de agentes activos, almacenamiento de grabaciones) ni integración con procesadores de pago (Stripe, PayU, Wompi Colombia). Sin billing, no hay negocio CCaaS.✗ CRÍTICOSin planes tarifarios ni diferenciación de funcionalidadesNo hay diferenciación de capacidades por plan (Basic/Pro/Enterprise). El marcador predictivo, las grabaciones ilimitadas y los reportes avanzados son candidatos naturales para planes premium que aumenten el ARPU.⚠ ALTOSin SLA definido ni mecanismo de garantíaPara vender CCaaS a empresas en Colombia o LATAM se necesitan SLAs de disponibilidad contractuales (99.9% mínimo), alertas de breach de SLA y créditos automáticos a clientes afectados. La arquitectura mono-nodo actual no puede garantizar ni 99%.ℹ OPORTUNIDADOportunidad: modelo open-coreLicencia MIT abre un modelo open-core: versión community gratuita + versión enterprise con multi-tenancy, SLA, soporte y funcionalidades avanzadas (IA, WhatsApp).
